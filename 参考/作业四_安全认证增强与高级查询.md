### 一、作业背景

课堂演示项目中，用户登录采用明文密码比对（`password.equals(user.getPassword())`），JWT Token 生成后无刷新机制，网关侧的 `SecurityFilter` 中 Token 校验逻辑也仅为示例代码。这些在实际生产环境中都存在安全隐患。

本次作业分两大部分：**Part A 安全认证增强** 和 **Part B 高级查询与统计**。同学们将在现有代码基础上进行渐进式改造，体验从"能跑通"到"更安全、更实用"的演进过程。

### 二、Part A：安全认证增强（60 分）

#### 2.1 密码加密改造（20 分）

**现状问题：** 当前 `UserServiceImpl.login()` 方法使用 `password.equals(user.getPassword())` 进行明文比对，数据库中存储的也是明文密码（如 `123456`）。

**改造要求：**

1. 在 `pom.xml` 中引入 `spring-security-crypto` 依赖（无需引入完整的 Spring Security）：
   ```xml
   <dependency>
       <groupId>org.springframework.security</groupId>
       <artifactId>spring-security-crypto</artifactId>
   </dependency>
   ```

2. 创建 `PasswordEncoderConfig` 配置类，注册 `BCryptPasswordEncoder` Bean：
   ```java
   @Configuration
   public class PasswordEncoderConfig {
       @Bean
       public BCryptPasswordEncoder passwordEncoder() {
           return new BCryptPasswordEncoder();
       }
   }
   ```

3. 修改 `UserServiceImpl.login()` 方法，将明文比对改为 BCrypt 校验：
   ```java
   // 旧代码：if (!password.equals(user.getPassword()))
   // 新代码：if (!passwordEncoder.matches(password, user.getPassword()))
   ```

4. 修改 `SysUserController.save()` 方法，在保存用户时对密码进行加密：
   ```java
   sysUser.setPassword(passwordEncoder.encode(user.getPassword()));
   ```

5. 提供数据库迁移 SQL，将现有明文密码批量更新为 BCrypt 加密格式（可编写一个一次性的 `DataMigrationRunner` 实现 `CommandLineRunner` 接口来完成）。

**验证标准：** 数据库中不再存在明文密码；使用旧密码 `123456` 登录仍然成功（因为加密后的值能被正确校验）。

#### 2.2 JWT Token 刷新机制（20 分）

**现状问题：** 当前 JWT Token 有效期固定 24 小时，过期后用户必须重新登录，体验不佳。

**改造要求：**

1. 新增 `/api/refresh-token` 接口，接收旧的（未过期或过期时间不超过 7 天的）Token，返回新的 Token：
   ```java
   @PostMapping("/api/refresh-token")
   public Result<LoginVo> refreshToken(@RequestHeader("Authorization") String authHeader) {
       // 1. 提取 Bearer Token
       // 2. 解析旧 Token（即使过期也尝试解析）
       // 3. 根据 userId 查询数据库确认用户仍然有效
       // 4. 生成新 Token 并返回
   }
   ```

2. 修改 `JwtUtil`，新增以下方法：
   - `parseExpired(String token)`：即使 Token 已过期也能解析出 Claims（捕获 `ExpiredJwtException` 并从中提取 Claims）。
   - `isExpired(String token)`：判断 Token 是否过期。
   - `getExpirationDays()`：获取配置的过期天数（从常量改为可配置）。

3. 刷新逻辑要求：
   - 如果 Token 未过期，直接签发新 Token（续期）。
   - 如果 Token 已过期但**不超过 7 天**，允许刷新。
   - 如果 Token 过期**超过 7 天**，拒绝刷新，返回 401。

**验证标准：** 使用 Apifox 先登录获取 Token → 等待 Token 过期 → 调用 refresh-token 接口获取新 Token → 用新 Token 访问受保护接口成功。

#### 2.3 网关安全过滤器完善（20 分）

**现状问题：** 网关 `SecurityFilter` 中 `validateToken()` 仅判断 Token 长度，`extractUserId()` 和 `extractUserName()` 返回固定占位符。

**改造要求：**

1. 在网关项目中引入 JWT 相关依赖（`jjwt-api`、`jjwt-impl`、`jjwt-jackson`）。

2. 在网关项目中创建 `JwtUtil` 工具类（或使用共享库），使用与 demo 服务**相同的密钥**来解析 Token。

3. 完善 `SecurityFilter` 中的三个方法：
   - `validateToken(String token)`：调用 `JwtUtil.validate()` 进行真实校验。
   - `extractUserId(String token)`：从 JWT Claims 中取出 `userId`。
   - `extractUserName(String token)`：从 JWT 的 `subject` 字段取出用户名。

4. 将 `@Component` 注解取消注释，使过滤器生效。

5. 利用 `IgnoreWhiteProperties` 配置类，将白名单路径从 Nacos 配置中心读取（而非硬编码），在 Nacos 中添加如下配置：
   ```yaml
   security:
     ignore:
       whites:
         - /api/login
         - /api/register
         - /api/refresh-token
         - /actuator
   ```

6. 修改 `SecurityFilter` 使用注入的 `IgnoreWhiteProperties` 替代硬编码的 `EXCLUDED_PATHS`。

**验证标准：** 通过网关访问受保护接口，不带 Token 返回 401；带有效 Token 正常访问；白名单路径无需 Token 即可访问。

### 三、Part B：高级查询与统计（40 分）

#### 3.1 用户统计接口（15 分）

在 `SysUserController` 中新增以下统计接口：

1. **用户总数统计** `GET /users/stats/total`：返回系统用户总数。

2. **按状态分组统计** `GET /users/stats/by-status`：返回各状态（启用/禁用）的用户数量，返回格式示例：
   ```json
   {
     "code": 200,
     "message": "操作成功",
     "data": [
       { "status": 1, "statusName": "启用", "count": 7 },
       { "status": 0, "statusName": "禁用", "count": 1 }
     ]
   }
   ```
   提示：使用 MyBatis-Plus 的 `selectCount()` 分别统计，或在 Mapper 中使用 `@Select` 自定义 SQL。

3. **最近注册用户** `GET /users/stats/recent?days=7`：返回最近 N 天内注册的用户列表。

#### 3.2 排序与多字段筛选增强（15 分）

修改现有的 `GET /users/listPage` 分页接口，增加以下能力：

1. **排序支持**：增加请求参数 `sortField`（排序字段）和 `sortOrder`（`asc` 或 `desc`），支持按 `age`、`createTime`、`username` 排序。

2. **年龄范围查询**：增加 `minAge` 和 `maxAge` 参数，使用 `LambdaQueryWrapper.between()` 或 `ge()/le()` 实现。

3. **状态筛选**：增加 `status` 参数，精确匹配。

4. 在 `SysUserQueryDto` 中新增上述字段，并在 Controller 中完善查询条件组装。

**示例请求：**
```
GET /users/listPage?pageNum=1&pageSize=10&sortField=age&sortOrder=desc&minAge=20&maxAge=30&status=1
```

#### 3.3 自定义 SQL 查询（10 分）

在 `SysUserMapper` 中新增以下自定义查询方法（使用 `@Select` 注解或 XML 映射文件均可）：

1. **按邮箱域名统计用户数**：
   ```java
   @Select("SELECT SUBSTRING_INDEX(email, '@', -1) AS domain, COUNT(*) AS count " +
           "FROM sys_user WHERE deleted = 0 " +
           "GROUP BY domain ORDER BY count DESC")
   List<Map<String, Object>> countByEmailDomain();
   ```

2. 在 Controller 中暴露接口 `GET /users/stats/email-domains` 调用上述方法。

### 六、参考提示

- BCrypt 加密后的密码长度固定为 60 个字符，请确保数据库 `password` 字段长度足够（当前 `VARCHAR(100)` 已满足）。
- `JwtUtil.parseExpired()` 的关键是捕获 `ExpiredJwtException`，其 `getClaims()` 方法仍然可以获取声明内容。
- 网关基于 Spring Cloud Gateway（WebFlux 响应式编程），不能使用 Spring MVC 的阻塞式 API，注意使用 `Mono` / `Flux` 编程模型。
- MyBatis-Plus 的 `LambdaQueryWrapper` 支持链式调用 `.orderBy(true, isAsc, column)` 实现动态排序。
- 建议在开发过程中使用 Nacos 本地 fallback 或直接在 `application.yml` 中配置默认值，避免因 Nacos 不可用影响调试。
